Task: Refine the Design Page and CSS Structure

We already have an initial /design page that collects UI examples from the old project CTWAnalysisWithClaudeCode. The current sections are:

Color Tokens
Space
Typography
Buttons
Form Fields
List Rows
Sections and Panels

This is a good starting point, but the structure is not yet clear enough to serve as a reliable design reference for future development. Also, the CSS is currently split between components.css and editor.css, but there is no dedicated token file yet.

The goal of this task is to refine the design page into a practical, living UI reference that prevents future UI work from inventing new colors, spacing, button styles, form layouts, and editor-specific patterns unnecessarily.

Goals
Create a clearer CSS architecture.
Extract reusable design tokens.
Reorganize the /design page into meaningful sections.
Ensure the design page uses real project classes, not fake demo-only styles.
Document reusable UI components and editor-specific UI patterns.
Add short usage rules for future UI work.
1. Add a dedicated tokens.css

Create a new CSS file for design tokens, for example:

static/css/tokens.css

This file should contain shared CSS custom properties for:

colors
spacing
typography sizes
border radii
borders
shadows, if used
z-index values, if useful

Example structure:

:root {
  /* Colors */
  --color-bg: ...;
  --color-surface: ...;
  --color-surface-hover: ...;
  --color-border: ...;

  --color-text: ...;
  --color-text-muted: ...;
  --color-accent: ...;
  --color-danger: ...;
  --color-success: ...;
  --color-warning: ...;

  /* Spacing */
  --space-1: 4px;
  --space-2: 8px;
  --space-3: 12px;
  --space-4: 16px;
  --space-6: 24px;

  /* Typography */
  --font-size-xs: ...;
  --font-size-sm: ...;
  --font-size-md: ...;
  --font-size-lg: ...;
  --font-size-xl: ...;

  /* Radius */
  --radius-sm: ...;
  --radius-md: ...;
  --radius-lg: ...;
}

After adding this file, update existing CSS to use these variables where reasonable. Avoid hardcoded colors, spacing values, font sizes, and radii when an existing token fits.

2. Clarify CSS file responsibilities

The current split between components.css and editor.css should be made more explicit.

Suggested structure:

static/css/
  tokens.css       // CSS variables: colors, spacing, type scale, radii, borders
  base.css         // optional: body, headings, links, code, global defaults
  components.css   // generic reusable UI components
  editor.css       // viewer/editor-specific UI patterns
  design.css       // styles used only by the /design page

It is okay if we do not create all files immediately. The important part is to avoid mixing responsibilities.

Suggested responsibilities:

tokens.css
- CSS variables only
- no component selectors

components.css
- buttons
- icon buttons
- form fields
- panels
- badges
- tabs
- notifications
- generic cards
- generic list rows

editor.css
- app shell
- topbar
- sidebar
- inspector
- canvas area
- canvas overlays
- region tree
- region list items
- XML/code preview blocks
- undo/history items

design.css
- layout and helper styles for the /design page only
- should not define new production UI components

4. Use real project classes on the design page

The design page should act as a living reference for the actual UI.

Avoid creating fake demo-only components such as:

<button class="design-demo-button">Button</button>

Prefer using real project classes:

<button class="btn btn-primary">Primary</button>
<button class="btn btn-secondary">Secondary</button>
<button class="btn btn-ghost">Ghost</button>
<button class="btn btn-danger">Delete</button>

The same applies to inputs, panels, badges, list rows, cards, tabs, and notifications.

The /design page may use design.css for its own layout, but the actual examples should use production classes from components.css and editor.css.

7. Add usage rules

Add a short usage rules section either directly on /design or in a new file such as:

STYLEGUIDE.md

Suggested rules:

# UI Styling Rules

- Use CSS variables from `tokens.css` for colors, spacing, typography, borders, and radii.
- Do not hardcode new colors if an existing token fits.
- Do not invent new button variants unless the design page is updated as part of the same change.
- Prefer existing component classes from `components.css`.
- Generic reusable UI belongs in `components.css`.
- Viewer/editor-specific UI belongs in `editor.css`.
- Styles used only to arrange the design page belong in `design.css`.
- New reusable UI patterns should be added to `/design`.
- The `/design` page should use real production classes, not fake demo-only component styles.
Acceptance Criteria

This task is complete when:

tokens.css exists and contains the main reusable CSS variables.
Existing CSS uses tokens where reasonable.
components.css and editor.css have clearer responsibilities.
The /design page is grouped into Foundations, Components, Editor / Viewer Patterns, Iconography, and Usage Rules.
The existing sections are preserved but better organized.
The design page uses real project classes for examples.
Buttons, form fields, list rows, panels, badges, tabs, and notifications show their intended variants and states.
Editor-specific patterns like region tree items, canvas selection states, inspector rows, and undo/history items are represented.
There is a short set of rules that future Claude Code tasks can follow before adding new UI styles.
