Canvas rendering
Two distinct visual vocabularies share the same SVG canvas. Sketch primitives use solid fills coloured by operation (add = teal, subtract = red) and are overlaid on top of the boolean result polygon. Editor regions use dashed, colour-coded outlines when deselected; a selected region switches to a solid outline and gains anchor blocks and resize handles.

Workspace pattern
Every activity workspace follows the same structure: one or two sidebars flanking a canvas, separated by draggable .sidebar-handle elements. Sidebars are fixed-width and hold .panel-section content in a scrollable inner div. The canvas fills remaining space.
How to read these trees
#topbar ID only — a one-of-a-kind shell element. CSS targets it by ID because it never repeats. .workspace-sidebar Class only — a shared pattern with no JS handle needed at this level. #ov-left.workspace-sidebar ID and class — the class does all the styling; the ID is a JS-only handle written as <div id="ov-left" class="workspace-sidebar">. The CSS file has no rule for #ov-left. #ov-left-scroll.workspace-scroll Same — class handles all layout including gap. The ID is a JS handle only; no CSS rule targets it.
Rule: if you see #id.class, the ID never appears as a selector in any CSS file. If you see a bare #id, the element is one-of-a-kind and CSS uses the ID directly.

Copy the pattern above. The ID sets the width; everything else comes from the class. The ov- / pt- prefix is a JS namespace — all workspaces live in the DOM simultaneously, so IDs must be unique. 6 — Usage Rules
Rules & decisions
Read before adding new CSS. If what you need is listed here, use what already exists instead of inventing something new.
Where does new CSS go?
Ask one question: Does this class know anything about PGM, CTW, regions, or the editor's page structure?
If no → components.css.  If yes → editor.css.
File Put here when… Examples
tokens.css Adding a new CSS custom property to :root --color-*, --space-*, --radius-*
components.css The class could be copy-pasted into a different project and still make sense. It has no dependency on the editor layout, game terminology, or workspace IDs. .action-btn, .badge, .field-input, .list-row, .step-dot, .toast, .console-output
editor.css The class only makes sense inside this app — it references a workspace layout, game concept (region, team, wool), or is a named structural element of the page shell. .region-row, .cat-header, .activity-btn, .draw-tool-btn, .detail-xml-pre, #topbar, #activity-rail, #dashboard-layout
design.css It exists only to lay out the /design page itself. Never needed in production. .ds-section, .ds-swatches, .ds-rules-table, #ds-nav

Token usage rules (checklist before adding CSS)
  • Is there a --color-* or --bg-* token that matches? Use it instead of a raw hex value.
  • Is the spacing value 4px, 8px, 12px, 16px, 20px, or 24px? Use the corresponding --space-N variable.
  • Is the radius 3px or 6px? Use --radius-sm or --radius-md.
  • Is the hover transition on colour or opacity? Use var(--transition-fast).
  • Is a new token genuinely needed? Add it to tokens.css and add a swatch here on /design.
Decisions already made — do not reinvent
  • Typeface: ui-monospace, monospace everywhere. No second font family.
  • Region category colours: use --cat-spawn / --cat-wool / --cat-build / --cat-other. Do not invent new category colours.
  • Row selection: --bg-selected for selected rows, --bg-selected-hover for hover. Do not use accent colour directly on rows.
  • Section titles: always .section-title (10px, uppercase, letter-spaced). Never make a panel title a different size.
  • Semantic labels: use .badge with a semantic variant (--success, --warning, --error, --neutral, --dim). Works both inline and standalone. Do not create separate tag or badge systems.
  • Notification system: exactly four types (topbar error, toast, canvas hint, panel warning). Adding a fifth requires updating this page.
  • Vis toggle: always .vis-btn. Appears on row hover by default; use .vis-btn--always to make it always visible.
  • Canvas colours: always use --canvas-* tokens (tokens.css). Never hardcode SVG fill, stroke, or overlay colours. Island polygons always use --canvas-island — the same gray in both the editor and sketch, with no per-island colour variation.
  • Sketch island tree: reuse .region-row, .region-type-icon, .region-label, .list-tag. The island level uses the layers icon. Shape children use their type icon from TYPE_ICON. No inline operation badges or override buttons — all per-shape actions go in the context menu.
  • Operation mode toggle: use .draw-tool-btn--op-add / --op-sub for the sketch add/subtract mode buttons. Separate the tool group and operation group with a .subbar-sep. Both groups use .draw-tool-btn as the base class.
  • New reusable patterns: add to components.css and demonstrate on /design before using elsewhere.
  • New editor-specific patterns: add to editor.css and demonstrate in the Editor Patterns section on /design.
  • IDs are for JavaScript, not CSS: workspace IDs (ov-left, pt-right, …) exist so JS can find elements. Do not use them as styling selectors — use .workspace-sidebar, .workspace-scroll, .workspace-canvas instead. Only override the one property that is genuinely unique per element (usually width).
